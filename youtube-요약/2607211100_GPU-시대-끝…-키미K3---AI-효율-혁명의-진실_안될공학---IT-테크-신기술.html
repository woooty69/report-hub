GPU 시대 끝?… 키미K3 | AI 효율 혁명의 진실

26.07.21.1118 min read

요약

키미 K3와 시장의 상반된 해석

안될공학 패치는 최근 엔비디아·마이크론 등 AI 반도체가 흔들린 배경으로 중국 문샷 AI의 키미 K3를 짚는다. K3는 전체 파라미터 2조 8천억, 최대 100만 토큰을 처리하며 896개 전문가 중 토큰마다 16개를 골라 쓰는 구조로, 이전 K2 대비 같은 연산을 실제 성능으로 바꾸는 효율이 약 2.5배 높아졌다고 한다. 이렇게 거대한 모델이면 ‘GPU가 몇 대나 필요할까’를 물어야 하는데, 시장에서는 정반대로 ‘이렇게 효율적인 AI가 계속 나오면 GPU를 예상만큼 안 사도 되는 것 아니냐’는 걱정이 나왔다. 같은 모델을 두고 GPU 수요가 줄어든다는 해석과 늘어난다는 해석이 동시에 나온 것이다. 여기에 K3가 이 성능을 얻는 과정에서 앤트로픽의 클로드를 활용한 것 아니냐는 의혹까지 붙었다(앤트로픽은 2월에 중국 AI 기업들이 부정 계정으로 클로드를 대량 호출해 능력을 추출하려 했다고 주장). 패치는 K3가 코딩·에이전트·지식 업무에서 이전 키미 모델을 크게 앞섰고 일부 평가에서 최신 폐쇄형 모델과 비교할 만한 결과를 냈다고 소개하되, 회사 자체 평가가 포함돼 아직 모든 조건이 독립적으로 재현된 것은 아니라고 단서를 단다. 그럼에도 이번 성과를 ‘파라미터를 크게 늘렸다’로만 설명하기 어려운 것은, 더 거대한 모델을 만들면서 계산은 더 희소하게 설계하고 성능은 최신 프런티어 모델에 가깝게 끌어올렸다는 점이라고 짚는다.

AI 연산 원리와 GPU·HBM이 필요한 이유

AI는 입력 문장을 토큰으로 나눠 숫자로 바꾸고, 여러 계층에서 수많은 파라미터와 곱·덧셈을 반복하는 거대한 행렬 계산으로 답을 만든다. GPU가 원래 화면의 수많은 픽셀을 동시에 처리하려 만들어진 병렬 구조라 이 계산에 딱 맞는다. 다만 GPU가 아무리 빨라도 계산할 숫자가 제때 도착하지 않으면 소용없어(계산기는 준비됐는데 문제지가 안 옴), 실행 중인 모델 가중치를 GPU 가까운 고대역폭 메모리(HBM)에 최대한 올려 둔다. 한 GPU에 다 안 들어가면 여러 GPU·여러 서버에 분산하므로, 2.8조 파라미터는 계산량뿐 아니라 ‘가까운 메모리에 펼쳐 놓을 공간’까지 요구한다. 학습은 가중치 외에 그래디언트·옵티마이저 상태·중간값까지 저장해야 하는 반면, 챗봇 질문 같은 추론은 완성된 파라미터로 답을 만드는 과정이라, 시장이 궁금한 것은 훈련에 GPU가 몇 대 들었느냐보다 ‘실제 서비스에서 답변 하나의 비용이 얼마나 내려갔느냐’다. 추론 중인 GPU 메모리에는 가중치뿐 아니라 입력을 처리하며 생기는 중간값, GPU끼리 데이터를 주고받는 통신 버퍼, 앞 문장을 기억하는 KV 캐시가 함께 들어간다. 거대한 모델에 짧은 질문이면 가중치가, 작은 모델이라도 긴 문서를 수백 명이 동시에 읽히면 KV 캐시가 더 큰 부담이 돼, MoE는 매번 계산에 참여하는 파라미터를, KDA는 긴 문맥에서 커지는 KV 캐시·어텐션 비용을 줄이는 서로 다른 부분을 겨냥한다.

MoE 구조 — 896개 전문가와 1.8%의 희소성

2.8조 파라미터를 질문마다 전부 계산하면 비용을 감당할 수 없어, K3는 MoE(전문가 혼합) 구조를 쓴다. MoE는 구글 스위치 트랜스포머·미스트랄·딥시크 V3·알리바바 Q3·이전 K2도 쓴 기술로 K3가 처음은 아니지만, K3는 이를 훨씬 크고 희소하게 밀어붙였다. Q3 235B는 전문가 128개 중 8개(6.3%), 딥시크 V3는 256개 중 8개(3.1%), K2는 384개 중 8개(2.1%)를 쓰는데, K3는 전문가 896개 중 16개(약 1.8%)만 고른다. 전체 모델은 커졌지만 실제 계산이 지나가는 길은 더 듬성듬성해진 셈이다. 다만 ‘전문가’라는 이름과 달리 수학은 수학 전문가에게 보내는 식으로 역할이 깔끔히 나뉜 게 아니라, 학습 과정에서 서로 다른 계산 패턴을 맡은 신경망 블록에 가까워 하나가 여러 패턴을 처리하기도 한다. 또 896개 중 16개만 쓴다고 나머지가 쉬는 게 아니라 입력·어텐션·라우터·공통 계층은 계속 작동하며, 16개 선택은 MoE 계층 안의 이야기이고 토큰이 여러 계층을 지날 때마다 라우터가 다시 고른다. 이 때문에 하드웨어에서는 896개 전문가 가중치를 여러 GPU의 HBM에 미리 나눠 올려 두는데, 선택된 전문가가 다른 GPU에 있으면 토큰을 그쪽으로 보내야 해 특정 전문가에 토큰이 몰리면 어떤 GPU는 바쁘고 다른 GPU는 기다리는 부하 불균형이 생긴다. 그래서 연산 성능만큼 HBM 대역폭·GPU 간 통신·전문가 배치가 중요해진다.

100만 토큰 문맥과 KV 캐시 — KDA·어텐션 레지듀얼

가중치만 봐도, 2.8조 파라미터를 16비트로 저장하면 약 5.6TB, 8비트 2.8TB, 4비트로 줄여도 약 1.4TB이고 여기에 KV 캐시·중간값·통신 버퍼가 별도로 든다. MoE는 매번 읽는 책 수를 줄일 뿐 도서관에 보관할 책 전체를 없애 주지 않는다. K3의 또 다른 큰 숫자는 100만 토큰 문맥으로, 책 여러 권·대규모 코드 저장소를 한꺼번에 넣는 규모다. AI는 긴 입력을 먼저 다 읽어 각 토큰의 정보를 만드는 ‘프리필’ 후 한 토큰씩 생성하는 ‘디코드’를 하는데, 매번 이전 문장을 처음부터 다시 읽지 않도록 핵심 계산값을 GPU 메모리에 저장한 것이 KV 캐시(책상 위 작업 메모)다. 대화·문서가 길어질수록, 특히 수백 명이 동시에 긴 자료를 읽히면 KV 캐시가 급증해 HBM 용량과 동시 사용자 수를 제한한다. K3는 이를 줄이려 KDA(키미 델타 어텐션)를 쓰는데, 과거 정보를 매번 다 펼치는 대신 제한된 크기의 상태에 계속 반영하는 선형 어텐션 계열이다(회의록을 쌓는 대신 하나의 작업 노트를 계속 고쳐 쓰는 방식). 다만 작은 상태에 압축하면 세부를 다시 찾을 때 손실이 생길 수 있어, 문샷은 KDA만 쓰지 않고 기존 어텐션과 결합한 하이브리드를 연구해 왔다(2025년 키미 리니어 연구에서 480억 파라미터 모델이 KV 캐시를 최대 75% 줄이고 100만 토큰 디코드를 약 6배 높였다고 보고, 단 이 수치를 2.8조 K3에 그대로 옮길 수는 없음). K3에는 뒤쪽 계층이 여러 깊이의 정보를 골라 가져오는 ‘어텐션 레지듀얼(어텐레스)‘도 들어간다(전화 게임에서 마지막 사람이 앞사람들의 기록까지 확인할 수 있게 한 셈). 문샷은 아직 K3의 실제 활성 파라미터·전문가 하나의 크기·공통 계층 규모를 공개하지 않아, ‘896개 중 16개’라는 숫자만으로 계산량이 정확히 16분의 1이라 말할 수 없다. 공통 계층은 계속 계산되고 전문가 하나의 크기에 따라 실제 연산이 달라지며, 계산에 참여하지 않은 전문가의 가중치도 사라지지 않아 전체 가중치는 어딘가에 준비돼 있어야 한다. K3는 이렇게 전문가를 희소하게 선택해 계산을 줄이고, KDA로 긴 문맥의 부담을 낮추며, 어텐션 레지듀얼로 깊어진 모델 안의 정보 흐름을 보완하는 기술들을 한곳에 모았다.

GPU 수요 논쟁과 제번스의 역설, 그리고 증류 의혹

‘정말 GPU가 덜 필요해질까’라는 물음에, 사용자 수와 사용량이 그대로면 답변당 계산·KV 캐시가 줄어 장비 수도 줄 수 있다. 하지만 효율이 높아진다고 총 수요가 반드시 주는 것은 아닌데, AI 비용이 내려가면 기업은 같은 일을 싸게 하는 데서 멈추지 않고 이전엔 비싸서 못 하던 업무(코드 저장소 전체 읽기, 에이전트를 몇 시간씩·여럿 동시 구동)까지 맡기기 때문이다. 사용자당 GPU 사용량이 절반으로 줄어도 전체 사용량이 10배 늘면 GPU는 오히려 더 필요해진다. 성능의 출처를 둘러싼 논쟁도 붙었는데, 앤트로픽은 2026년 2월 23일 딥시크·문샷·미니맥스가 약 2만 4천 개 부정 계정으로 클로드를 100만 회 넘게 호출했고 문샷은 후반에 클로드의 추론 과정을 추출·재구성하려는 표적화된 시도를 했다고 주장했다. 핵심은 ‘증류’라는 기술 자체(상위 모델의 답변으로 다른 모델을 훈련하는 널리 쓰이는 방식)가 아니라, 규정·접근 제한을 우회해 경쟁사 모델을 비정상적으로 대량 호출해 학습 데이터를 확보했느냐다. 이렇게 얻은 데이터는 정답뿐 아니라 문제를 나누는 순서·도구 선택·실패 후 수정 경로까지 담겨 후학습 시간·비용을 줄일 수 있다(딥시크도 2025년 초 오픈AI API 대량 인출 논란). 다만 그 데이터가 K3에 직접 쓰였는지는 확인되지 않아 ‘심증만 있고 물증은 없는’ 상태이며, API로 얻는 것은 출력이지 원본 가중치가 아니어서 2.8조 파라미터 모델을 만들려면 여전히 자체 기반 모델·연산 인프라·학습 기술이 필요하다. 시장의 더 직접적인 문제는 ‘투자금을 얼마나 빨리 회수하느냐’로, 중국 오픈 모델이 최상위 모델에 가까운 능력을 더 싸게 제공하면 고객은 가장 어려운 작업만 폐쇄형에 맡기고 일반 업무는 저렴한 오픈 모델로 돌려, GPU가 필요 없어지는 게 아니라 막대한 투자를 한 기업이 높은 수익을 오래 유지할 수 있느냐가 불확실해진다. 결론적으로 K3는 딥시크·구글 터보컨트 때처럼 ‘효율 개선=메모리 수요 감소’로 단정하기 어렵다는 것을 다시 보여주며(절약한 메모리는 더 긴 문맥·더 많은 동시 사용자·복잡한 에이전트에 쓰임), GPU를 필요 없게 만든 모델이 아니라 막대한 AI 인프라 투자가 그만큼의 성능·수익으로 돌아오는지를 더 까다롭게 살피게 만든 모델에 가깝다고 정리한다. 봐야 할 것은 ‘AI가 메모리를 덜 쓰게 됐느냐’가 아니라 ‘절약한 메모리로 AI가 얼마나 더 많은 일을 하게 되느냐’다. 패치는 문샷이 세부 기술 자료를 공개하면 벤치마크 순위보다 먼저 볼 숫자로, 토큰마다 실제 계산되는 파라미터 수, 모델 전체를 올리는 데 필요한 HBM, 100만 토큰에서 첫 답변까지 걸리는 시간, 896개 전문가를 여러 GPU에 나눴을 때의 통신량과 특정 전문가에 토큰이 몰릴 때의 성능 저하, 그리고 학습에 외부 모델 출력이 활용됐는지에 대한 설명을 확인하라고 조언한다. 딥시크 V3·R1도 MoE·MLA로 계산량과 KV 캐시 부담을 낮췄을 때 인프라 과잉 투자 우려로 반도체주가 흔들렸고, 구글 터보컨트도 장문맥 KV 캐시를 낮은 비트로 압축해 비슷한 반응을 낳았지만, 절약분을 더 긴 문맥이나 더 많은 동시 처리로 돌려쓰는 탓에 효율 개선이 곧바로 전체 메모리 수요 감소로 이어지진 않았다.
