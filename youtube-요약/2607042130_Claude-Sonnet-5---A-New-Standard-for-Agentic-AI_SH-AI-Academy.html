Claude Sonnet 5: A New Standard for Agentic AI?

Claude Sonnet 5: A New Standard for Agentic AI?

요약

왜 Sonnet 5가 중요한가 — ‘에이전틱 격차’를 좁힘

  • Anthropic이 2026년 6월 30일 Claude Sonnet 5 출시. 점진적 개선이 아니라 상당한 도약.
  • 지난 1년간 에이전틱 AI는 **이분화(bifurcation)**되어 있었음: Sonnet급(3.5～4.6)은 빠르고 비용 효율적인 ‘일꾼’이지만, 진짜 자율 실행(다단계 계획, 여러 도구 호출, 디버깅 루프, 자기교정)이 필요하면 Opus 4.8을 써야 했음.
  • Sonnet 5는 그 격차를 메우는 모델 — 낮은 비용으로 Opus 4.8에 근접, 에이전틱 벤치마크에서 이전 어떤 Sonnet보다 넓은 비용-성능 곡선을 커버하며 높은 effort에서는 일부 작업에서 Opus 4.8과 대등.

얼리액세스 파트너가 관찰한 3가지 새 행동

  • 끝까지 완수한다(It finishes): 이전 Sonnet이 중간에 멈추던 복잡한 작업을 끝냄. 예 — Salesforce 계정 티어 업데이트 후 기업 연락처에 출시 공지 발송하는 2단계 워크플로가 예전엔 절반에서 멈췄으나 Sonnet 5는 엔드투엔드로 완료.
  • 시키지 않아도 자기 출력을 검증한다: Rust 엔지니어가 버그 조사를 시키자, 요청하지 않았는데도 재현 테스트 케이스를 작성 → 수정 구현 → 수정본을 임시 보관(stash)하고 테스트를 재실행해 버그가 되살아나는지 확인까지 단일 패스로 수행. 이 자기주도적 검증 루프가 단일턴 도구 사용과 에이전틱 능력을 가르는 지점.
  • 레거시(brownfield) 코드에 강함: 경쟁 상태·숨은 테스트·미문서화 동작이 있는 오래된 코드베이스에서 증상만 패치하지 않고 실제 근본 원인까지 추적.
  • (추가) 적절한 거부(appropriate refusals): 안전하지 않은 요청을 깔끔하고 일관되게 거부 — 강력한 도구를 다수 사용자에게 줄 때 중요.

벤치마크 결과

  • SWE-bench Verified(사람 개입 없이 실제 GitHub 이슈 자율 해결): Sonnet 4.6 대비 명확한 개선, Opus 4.8과의 격차 축소.
  • Humanity’s Last Exam (with tools)(PhD급 난이도): Sonnet 4.6은 46.8%였고(출시 시 채점 모델 변경으로 갱신됨) Sonnet 5가 개선.
  • 에이전틱 2종 — Browse Comp(에이전틱 웹 검색), OSWorld Verified(실제 컴퓨터 GUI 자동화): 비용-성능 곡선으로 표시. 모든 effort 레벨에서 Sonnet 4.6 대비 엄격한 개선, extra-high effort에서 두 벤치마크 모두 Opus 4.8과 대등. (OSWorld는 방법론 개정으로 Sonnet 4.6 기준선이 78.5%로 조정됨.)

개발자 필독 — 코드가 깨질 수 있는 3가지 동작 변경

  1. Adaptive thinking이 기본 ON: Sonnet 4.6에서는 thinking 필드가 없으면 추론 없이 실행됐으나, Sonnet 5는 동일 요청이 기본적으로 adaptive thinking으로 실행(작업 복잡도에 따라 모델이 사고 여부·양을 결정). 끄려면 thinking type: disabled를 명시해야 함.
  2. 수동 extended thinking 완전 제거: thinking type: enabled + budget_tokens 패턴(4.6에서 이미 deprecated)이 완전히 사라짐 → Sonnet 5에서 400 에러. 대신 adaptive thinking + effort 파라미터 사용.
  3. 샘플링 파라미터 오버라이드 제거: temperature, top_p, top_k를 비기본값으로 설정하면 400 에러(Opus 4.7·4.8에서 이미 제거). API 호출에서 제거하고 대신 시스템 프롬프트로 동작을 유도.
  • 중요 실무 노트: adaptive thinking이 기본 실행되고 thinking 토큰이 max_tokens 예산에 포함되므로, 4.6에서 출력용으로만 잡아둔 max_tokens 한도가 Sonnet 5에선 너무 빡빡할 수 있음 → 재검토 필요.

새 토크나이저

  • Opus 4.7과 유사한 토크나이저 변경. 같은 입력 텍스트가 Sonnet 4.6 대비 약 30% 더 많은 토큰 생성(콘텐츠 유형에 따라 배수 1.0～1.35 — 코드·산문·기타 형식이 다르게 토큰화).
  • 실무 영향: 토큰 카운트 API 반환값이 커짐. 4.6 기준으로 캐싱/계산한 토큰 수는 무효 → 재계산. 컨텍스트 윈도우는 100만 토큰이지만 토큰당 커버 텍스트가 줄어 같은 윈도우에 담기는 텍스트가 적어짐. 토큰당 가격은 동일해도 요청당 비용은 달라질 수 있음(토큰 수가 늘어서).
  • 변하지 않는 것: 요청·응답·스트리밍 이벤트 구조 동일 → 토크나이저 때문에 코드 변경 불필요.

가격 및 가용성

  • 토큰당 가격 동일: 표준 입력 100만 토큰당 3,출력15.
  • 도입가(2026-08-31까지): 입력 2/출력10 — 새 토크나이저의 토큰 증가를 상쇄해 4.6 대비 대략 비용 중립이 되도록 설계. 9월 1일부터 표준가 적용 → 그 전에 새 토큰 수 기준으로 비용 재산정 필요.
  • 가용성: 모든 Claude 플랜(무료·프로·맥스·팀·엔터프라이즈)의 새 기본 모델. Claude Code, API(Claude 플랫폼 직접, AWS Bedrock의 신규 Claude 플랫폼, Google Vertex AI, MS Foundry 프리뷰)에서 출시일부터 제공. ZDR 계약 조직은 제로 데이터 보존 지원.
  • 스펙: 컨텍스트 100만 토큰(기본이자 최대, 더 작은 변형 없음), 최대 출력 128,000 토큰, 모델 ID claude-sonnet-5.
  • 주의: 출시 시점 priority tier 미지원(4.6엔 있었음). Anthropic은 adaptive thinking·고 effort로 인한 토큰량 증가에 대비해 챗·Cowork·Claude Code·플랫폼 전반의 rate limit을 상향.

effort 파라미터 — 두 개의 다이얼

  • effort가 응답 전 사고량을 제어: low(빠르고 저렴) / medium(대부분 작업의 adaptive 기본값, 복잡도 비례) / high(다수 작업에서 Opus 4.8 근접) / extra-high(Browse Comp·OSWorld에서 Opus 4.8과 대등).
  • 이제 레버가 둘: 모델 선택(Sonnet 5 vs Opus 4.8) × effort 레벨. 두 축으로 비용·성능을 연속적으로 조정 — Sonnet 품질과 Opus 품질 사이의 ‘절벽’이 사라짐.
  • 예산 관점(도입가): Sonnet 5 출력 10/MvsOpus4.8출력25/M — X-high effort로 Opus에 대등해도 토큰당은 여전히 훨씬 저렴. 손익분기는 작업 분포에 따라 다름. adaptive 기본이라 대부분 budget_tokens를 수동 설정할 필요 없음.

안전성 평가

  • Anthropic 자동 행동 감사(기만·오용 협조 등 광범위 시나리오)에서 Sonnet 5가 4.6보다 낮은 점수 = 더 안전. 환각(hallucination)·아첨(sycophancy) 비율도 낮음. 악성 요청 거부 개선, 프롬프트 인젝션 저항 강화(에이전틱 파이프라인에서 실질적 위협이라 의미 큼).
  • 단서: Sonnet 5의 misalignment 비율은 여전히 Opus 4.8·Claude Mythos 프리뷰보다 높음 — 더 유능한 모델이 정렬도 더 좋은 흔한 패턴 속에서 Sonnet 5는 4.6과 Opus급 사이에 위치.
  • 사이버 세이프가드(신규): Sonnet 티어 최초로 실시간 사이버 안전장치 기본 활성화(위험한 사이버보안 요청을 실시간 탐지·차단, Opus 4.7·4.8과 동일하나 Fable 5보다는 덜 엄격). 거부는 HTTP 200 + stop_reason: refusal로 반환(4xx 아님) → 에러 핸들링이 stop_reason을 확인해야 함. 사이버 능력 테스트에서 Firefox 취약점 익스플로잇 완전 성공 0%(부분 성공률은 4.6보다 약간 높으나 일반 지능 향상에 기인).

마이그레이션 체크리스트 (5단계)

  1. 모델 ID를 claude-sonnet-4-6 → claude-sonnet-5로 변경.
  2. 샘플링 오버라이드 제거(temperature·top_p·top_k → 400 에러). 대신 시스템 프롬프트로 유도.
  3. extended thinking 설정 마이그레이션: thinking type: enabled + budget_tokens → adaptive로 교체(필드 생략 시 기본 adaptive, 명시하려면 type: adaptive, 끄려면 type: disabled).
  4. 토큰 예산 재계산: 새 토크나이저로 대표 프롬프트를 토큰 카운트 API에 돌려 정확 수치 확보, 출력 길이에 근접했던 max_tokens 재검토.
  5. 거부 핸들링 업데이트: 사이버보안 관련 가능성이 있으면 모든 응답에서 stop_reason: refusal 확인(HTTP 200으로 옴). 예외/비200만 잡으면 사이버 거부를 놓침.
  • 변경 불필요: 도구 정의, 응답 구조, 스트리밍 포맷, 시스템 프롬프트, 응답 파싱 — 4.6과 동일. (assistant message prefilling은 4.6에서도 이미 미지원.)

7가지 핵심 요점

  1. Sonnet 5가 에이전틱 격차를 닫음 — Sonnet/Opus 능력 차이는 절벽이 아니라 effort로 조정하는 다이얼. high·extra-high에서 일부 작업 Opus 4.8과 대등.
  2. Adaptive thinking 기본 ON — 수동 budget_tokens 불필요, 끄려면 명시적 disable.
  3. 세 가지가 400 에러 — 수동 extended thinking(budget_tokens), 샘플링 오버라이드, assistant message prefilling.
  4. 새 토크나이저로 같은 텍스트가 ～30% 더 많은 토큰 — 재계산·max_tokens 재검토·9월 이후 비용 재추정.
  5. 컨텍스트·출력 한도 대폭 확장 — 200K→100만 토큰, 8K→128K 출력. 새로운 작업 범주를 여는 변화.
  6. 사이버 세이프가드 기본 ON(Sonnet 최초) — 거부는 HTTP 200 + stop_reason refusal.
  7. 그럼에도 4.6의 드롭인 교체 — 핵심 마이그레이션은 3가지 코드 변경(extended thinking·샘플링 제거, 모델 ID 변경, 토큰 재계산). 도입가 기간(8/31까지)이 지금 이전하기 좋은 시점.
